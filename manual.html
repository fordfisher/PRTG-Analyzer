Section 1

What is PyPRTG_CLA?

PyPRTG_CLA (Python PRTG Core Log Analyzer) is a standalone, local web application built specifically for PRTG Network Monitor. It takes a single Core.log file as input, parses every relevant line with deterministic regex patterns, evaluates the extracted data against a set of 11 hardcoded best-practice rules, computes a 0–100 health score, and presents the results as an interactive browser-based dashboard.

Zero AI. Zero guessing. Every finding, every score delta, every recommendation is produced by explicit, auditable logic that was written by engineers who know PRTG inside out. If the data is not in the log, it is not shown. Period.

Who is it for?

  • Paessler Support Engineers — instantly assess a customer instance from a single log file, without access to the live PRTG instance.
  • NOC Operators — quickly understand the health of monitored PRTG servers.
  • PRTG Administrators — proactively review their own instance against Paessler's official sizing guidelines.

What does it analyze?

Everything in the main analysis comes directly from the log. When a Status Data file is uploaded, an additional Status Snapshot shows live metrics from the bundle.

From the log:

  • Server name, PRTG version, license owner, OS, CPU, RAM, storage paths, timezone
  • Total sensors, probes, devices, groups, channels, users
  • Sensor impact distribution (five levels: Very Low → Very High)
  • Sensor refresh-rate (interval) distribution
  • Per-probe sensor counts, ERP (Estimated Requests Per second), and impact breakdown
  • All ERRR-level log errors, normalized and deduplicated into patterns
  • Restart history and startup milestones
  • Thread runtime statistics
  • CPU splitting configuration
Section 2

System Requirements

  • Operating System: 64-bit Windows 10 or later
  • Browser: Microsoft Edge, Google Chrome, or Mozilla Firefox (modern, evergreen release)
  • Network: Localhost access — TCP port 8077 must not be blocked by a host-based firewall
  • Disk: A small amount of temporary disk space for the result cache (written to the OS temp directory)
  • RAM: No special requirement — parsing is streaming and memory-efficient even for files of several hundred MB
No Python required. The EXE is fully self-contained. No runtime, no virtual environment, no additional installer is needed on the target machine.
Section 3

Installation & First Launch

There is no installation step. The entire application ships as a single file:

PyPRTG_CLA.exe

Starting the application

  1. Locate PyPRTG_CLA.exe and double-click it.
  2. If Windows SmartScreen shows an "unrecognized app" warning:
    • Click More info.
    • Then click Run anyway.
    The binary is unsigned. No data is sent anywhere — this warning is expected for unsigned executables.
  3. A console window opens. The first launch may take 5–15 seconds while PyInstaller unpacks the embedded Python runtime into a temporary folder. Subsequent launches on the same machine are faster.
  4. Wait until the console shows:
    Uvicorn running on http://127.0.0.1:8077
  5. Open your browser and navigate to:
    http://127.0.0.1:8077

Stopping the application

  • Close the console window, or
  • Click into the console and press Ctrl + C to stop the server gracefully.
Custom port. If port 8077 is taken, set the environment variable PRTG_ANALYZER_PORT to a different number before launching.

Uploading Core.log and Status Data

Core.log (required)

FormatExtensionNotes
Plain text log.logStandard PRTG Core.log
Gzip-compressed log.gzDecompressed transparently on upload
Plain text (renamed).txtAlso accepted

File encoding is auto-detected: UTF-8, UTF-16, and latin-1 are all handled.

Status Data (optional)

You can optionally upload a PRTG Status Data file from the same support bundle. This adds a Status Snapshot panel showing live sensor counts and metrics from when the bundle was created (instead of from the last PRTG restart).

FileFormatWhat it provides
PRTG Status Data.htmHTMLFull metrics: CPU load, RPS, slow request ratio, impact distribution
Status Data.htmJSONSensor status counts: up, warning, down, paused, unknown

Both formats are auto-detected. If no Status Data file is uploaded, the app works exactly as before.

How to upload

  1. (Optional) Click Upload Status Data and select your .htm file from the support bundle. The button will show the selected filename.
  2. Upload Core.log: Drag and drop your Core.log onto the upload area, or click it to open a file browser.
  3. A live progress bar shows parsing status in real time. Large files (hundreds of MB) are processed with a streaming parser — memory usage stays low.
  4. When analysis completes, the dashboard appears automatically with all tabs and charts populated.

Instant re-analysis from cache

Every uploaded file is identified by its SHA-256 hash. If you upload the same file again (even after closing and reopening the app), the result is served from cache instantly — no re-parsing is needed. The cache is keyed to the analyzer version, so upgrading to a new version of PyPRTG_CLA automatically invalidates old cached results and triggers a fresh analysis.

Where does PRTG store Core.log? On a standard PRTG installation, the file is located at: C:\ProgramData\Paessler\PRTG Network Monitor\Logs\Core\Core.log
Section 5

Tab: Overview

The Overview tab is the landing view after a successful analysis. It provides a full at-a-glance picture of the PRTG instance.

View toggle (Now / Historical)

If you uploaded a Status Data file, a View: Now | Historical toggle appears above the dashboard. Use it to switch between:

  • Now — Shows the Status Snapshot panel (live data from the bundle). The time-window slider is hidden.
  • Historical — Shows the time-window slider. The Status Snapshot panel is visible when the slider is set to "All".

Status Snapshot panel (when Status Data was uploaded)

This panel shows metrics from the Status Data file (bundle creation time). For HTML format: total sensors, CPU load, requests/second, slow request ratio, and sensor impact by level. For JSON format: total sensors, up/warning/down counts (color-coded), and PRTG version. When available, values are compared to the last boot from the Core.log.

Global Snapshot panel

Displays the most critical identifiers extracted directly from the log:

  • Server name — the hostname of the PRTG server
  • PRTG version — e.g. 26.1.1
  • License owner — the licensed organization name
  • System ID — the internal PRTG SystemID
  • Total sensors / probes
  • CPU splitting status — whether CPU splitting is active
  • Health score — the computed 0–100 score with color-coded verdict

Instance Health panel

  • Health verdict — one of: Healthy, Optimization Needed, or Critical
  • Log span days — calendar range covered by the log file
  • Restart count — total number of PRTG Core restarts detected
  • Error and warning pills — quick count of ERRR and warning lines

Metrics grid (4 cards)

CardValues shown
Load & ERP Calculated requests per minute, total ERP from log, busiest probe name
Stability Restart count, total warning count
License & OS License owner, OS version string

Overview charts

Six interactive ECharts visualizations are rendered in the Overview tab:

Stability Radar

A five-axis radar chart showing the instance's stability profile across: Errors, Restarts, Startup, Threads, and Load. Each axis is normalized so a perfect score produces a full pentagon, and degraded areas shrink the shape inward.

ERP Load Orbit (Gauge)

A gauge showing the calculated scanning load as a percentage of "safe capacity" (10,000 requests per minute per probe). Values above 100% indicate a potentially overloaded scanning schedule.

Memory Utilization (Gauge)

A gauge showing RAM used as a percentage of total installed RAM, calculated from the values extracted directly from the log.

Sensor Impact Levels (Donut chart)

The global distribution of sensors across five impact levels: Very Low, Low, Medium, High, and Very High. A large share of Medium/High/Very High sensors is a known contributor to high CPU load in PRTG.

Error Activity Over Time (Line chart)

Plots daily error volume across the log's time span, allowing you to spot sudden error spikes or periods of unusual instability.

ERP Hot Probes (Horizontal bar chart)

The top 8 probes ranked by their ERP (Estimated Requests Per second) value as reported in the log. Identifies which probes are driving the most scanning load.

Section 6

Tab: Findings

The Findings tab lists every rule that triggered during the analysis of this log. Only rules that actually fire are shown — if your instance has no issues in a particular area, that rule is simply absent from the list.

Each finding card shows:

  • Title — a plain-language description of the issue, color-coded by severity
  • Rule ID — the internal rule reference (e.g. RULE_2_RAM)
  • Score delta — how many points this finding deducted from the health score
  • Evidence — the actual values from the log that triggered this rule (e.g. "Current RAM: 12 GB, Required: 16 GB")
  • Recommendation — a concrete action, with tone scaled to severity:
    • Slightly over threshold → "Consider…"
    • Moderately over threshold → "We recommend…"
    • Severely over threshold → "Immediate action required…"
All thresholds and recommendations are aligned with Paessler's official PRTG setup guide. There is no machine learning, no heuristic guessing — every recommendation maps to a specific published guideline.

See Section 13 — Health Score & Rules Engine for the complete list of all 11 rules and their exact thresholds.

Section 7

Tab: Top Errors

The Top Errors tab shows the most frequently occurring error patterns extracted from all ERRR-level lines in the log.

How errors are processed

Raw error lines are normalized before counting. The normalization replaces variable parts with placeholders so that many instances of the same underlying error are correctly grouped into a single pattern:

Variable partReplaced with
GUIDs (e.g. 550e8400-e29b-...){<guid>}
Thread IDsTId <id>
Hexadecimal values0x<hex>
Numbers with 2 or more digits<n>

This normalization is purely deterministic — the same raw log always produces the same pattern set. The analyzer stores up to 500 unique error patterns and surfaces the top 10 in this tab.

License errors are excluded

License activation and key-validation errors are explicitly excluded from all error counts and pattern lists. These errors are common and expected in many PRTG log files and would otherwise distort the error picture. Only genuine operational errors are counted.

What each error card shows

  • Rank — position by frequency (#1 = most frequent)
  • Occurrence count — exact number of times this pattern appeared
  • First seen / Last seen — timestamps bounding this error's presence in the log
  • Normalized pattern — the deduplicated error text
  • Up to 5 sample lines — actual raw log lines to aid diagnosis

Toolbar controls

  • Show Top 5 / Show Top 10 — toggle between seeing the 5 or 10 most frequent patterns
  • Copy — copies the full error list to the clipboard for pasting into a ticket or report
Section 8

Tab: Sensors & Probes

This tab provides a deep dive into how scanning load and sensor risk are distributed across the instance.

Sensor Refresh Rate Distribution (bar chart)

A vertical bar chart showing how many sensors are set to each scanning interval (e.g. 30 seconds, 1 minute, 5 minutes, 1 hour, …). Short intervals on many sensors are the primary driver of high scanning load (ERP). The chart labels intervals in human-readable form.

ERP Relative Load by Sensor Type (horizontal bar chart)

Shows the top 20 sensor types ranked by their weighted scanning contribution (sensor count × 60 ÷ interval). This makes it easy to spot which sensor technologies are disproportionately contributing to load.

Per-probe impact distribution charts

For each of the top 10 probes (by sensor count), a bar chart shows the number of sensors at each of the five impact levels: Very Low, Low, Medium, High, and Very High. If more than 10 probes exist, a note indicates how many additional probes are present but not charted individually.

Why impact levels matter: PRTG's engine processes sensors differently depending on their impact level. A high share of Medium/High/Very High sensors increases CPU and memory usage disproportionately.

Probes & Sensor Distribution table

A summary table listing every probe with its:

  • Name — probe display name from the log
  • ID — internal PRTG probe ID
  • Sensor count — number of sensors assigned to this probe
  • ERP — the probe's Estimated Requests Per second, as reported by PRTG in the log
Section 9

Tab: Timeline

The Timeline tab maps the history of the PRTG Core process over the log's time span, helping you understand restart patterns and correlate instability with error spikes.

Timeline Events list

Each event is shown with its timestamp, a type label in brackets, and a descriptive text:

  • [restart] — a PRTG Core restart detected via a Logger initialized line
  • [milestone] — a significant startup milestone line logged by PRTG

Timeline scatter chart

Events are plotted on a horizontal time axis. Restart events appear on one row and milestones on another, making it immediately clear whether restarts cluster at a particular period or are evenly distributed.

Interpreting restart density: More than 10 restarts across the log span triggers the high restart frequency finding. More than 5 restarts triggers the elevated frequency warning. See Section 13 for exact thresholds.

Section 11

Timeframe Filter

The Global Time Window Slider appears above the tabs once a log has been analyzed. It lets you narrow the entire analysis — score, findings, errors, and charts — to a specific slice of the log's history.

How it works

PRTG logs a Logger initialized line every time the Core process starts. The analyzer uses these lines to divide the log into restart segments. The slider lets you choose how many of the most recent segments to include, from the very last one up to all segments.

  • Drag the slider to 1 to analyze only the current (most recent) restart cycle.
  • Drag it to the maximum to analyze the full log (the default).
  • Restart timestamp markers are shown as dots on the slider track so you can see exactly when each segment started.

What changes with the timeframe

  • Health score is recomputed using only the data from the selected segments
  • All findings are re-evaluated
  • Error counts (total, last 24h, since restart) are filtered to the selected window
  • Top error patterns reflect only the selected time window
  • System info fields show the configuration as it was at the start of the selected segment
Pro tip: Set the timeframe to the last 1 restart to assess the current state of the instance in isolation, without historical noise from months of old data.

Performance

Each timeframe result is cached in memory in the browser. Switching between timeframe values is instant once they have been computed — no repeated server round-trips.

Section 12

Export

The Export tab offers two download options, both of which respect the currently active timeframe filter.

JSON Export

Downloads the complete analysis result as a structured JSON file. The JSON includes:

  • All parsed fields from the log (core object — server info, sensors, probes, errors, threads, etc.)
  • The computed score (0–100)
  • The full findings list with rule IDs, evidence, and recommendations
  • The refresh_rate_distribution array
  • The calculated_requests_per_min value
  • The timeline events
  • A metadata block with analyzer version and file hash

The JSON is suitable for custom scripts, integrations with other tools, or archival.

HTML Report

Downloads a fully self-contained standalone HTML file — no internet connection is needed to open it. The report contains:

  • A header identifying the report as PyPRTG_CLA Enterprise Report — [Server Name]
  • Global Snapshot card
  • Key Signals card
  • Global Impact Distribution donut chart (ECharts, embedded)
  • Refresh Rate Distribution bar chart (ECharts, embedded)
  • Full Findings & Recommendations list
  • Top Errors list
License key protection. The HTML report automatically masks the PRTG license key: only the first 6 and last 4 characters are shown (e.g. 001001…N47D). The full key is never written into the exported file, making reports safe to share with customers or attach to support tickets.
Section 13

Health Score & Rules Engine

The health score starts at 100 and has points deducted by each rule that fires. The final score is clamped to the range 0–100. The score translates to a verdict:

≥ 80
Healthy
60 – 79
Optimization Needed
< 60
Critical

The 11 Hardcoded Rules

All rules are static, explicit code. They were written based on Paessler's published PRTG sizing guidelines and support knowledge base. There are no configurable thresholds, no weights to tune, and no machine learning involved. What you see in the tool is exactly what the code computes.

Rule ID Title Trigger condition Severity Score
RULE_1_CPU CPU cores below recommended CPU cores < 95% of required tier Critical −10
RULE_2_RAM RAM below recommended tier RAM < 90% of required tier Critical −10
RULE_4_OS Deprecated Windows Server version OS year is 2008 or 2012 Critical −10
RULE_4_OS Aging Windows Server version OS year is 2016 Warning −5
RULE_7_ERRORS Errors detected in Core.log > 0 operational errors; escalates if > 50 distinct patterns Warning / Critical −5 / −8
RULE_11_ERP High scanning load Calculated req/min > 15,000 Critical −10
RULE_11_ERP Elevated scanning load Calculated req/min > 10,000 Warning −6
RULE_13_DRIVES Data and System paths on same drive Data drive letter == System drive letter Warning −5
RULE_14_RESTARTS High restart frequency Total restarts > 10 Critical −5
RULE_14_RESTARTS Elevated restart frequency Total restarts > 5 Warning −3
RULE_15_THREADS Very long-running threads Max thread runtime > 300 s Critical −5
RULE_15_THREADS Long-running threads Max thread runtime > 120 s Warning −3
RULE_16_IMPACT High share of medium/high impact sensors (Medium + High + Very High) / total > 30% Warning −4
RULE_17_CHANNELS Many sensors exceed 50 channels Sensors with >50 channels > 10 Warning −3
RULE_18_CPU_SPLITTING CPU splitting active with >8 CPUs CPU splitting enabled AND total CPUs > 8 Warning −4

CPU sizing tiers (hardcoded)

Sensor countMinimum CPU cores required
< 2,5004 cores
2,500 – 4,9998 cores
5,000 – 9,99912 cores
≥ 10,00016 cores

The rule fires when the actual core count is below 95% of the required tier (a 5% tolerance band is applied to avoid penalizing borderline cases).

RAM sizing tiers (hardcoded)

Sensor countMinimum RAM required
< 2,5008 GB
2,500 – 4,99916 GB
5,000 – 9,99932 GB
≥ 10,00064 GB

The rule fires when actual RAM is below 90% of the required tier.

Scanning load calculation

The calculated requests per minute is derived directly from the sensor interval distribution parsed from the log, using this formula applied to each interval bucket: 

requests_per_min += (sensor_count × 60) ÷ interval_seconds

This is independent of the ERP value in the log, which is an instantaneous measurement. The calculated value reflects the theoretical steady-state load based on the configured intervals.

Section 14

Reliability & Data Integrity

PyPRTG_CLA is built with a single overriding principle: every value it displays must be traceable back to a specific line in the uploaded log file. This section explains the measures in place to guarantee that.

1. Zero AI — fully deterministic

The analyzer contains no language models, no machine learning, and no probabilistic inference of any kind. Every parsed value is extracted by an explicit regular expression pattern. Every score deduction is triggered by an exact numerical comparison. The same log file will always produce exactly the same result, on any machine, at any time.

2. If it's not in the log, it's not shown

Fields that cannot be parsed from the log are left empty or omitted from the output entirely. The analyzer never fills in defaults, never estimates, and never guesses values that are absent from the source data.

3. License error exclusion prevents false positives

License activation and key-validation errors are explicitly filtered out before counting ERRR lines. These errors are ubiquitous in many PRTG Core.log files and are not indicators of operational problems. Including them would inflate the error count and potentially trigger the error-severity rule unfairly.

4. Deterministic error normalization

Error deduplication is performed by replacing variable parts (GUIDs, thread IDs, hex values, long numbers) with fixed placeholder tokens via regex substitution. This grouping logic is applied identically to every error line — the resulting pattern set is reproducible and auditable.

5. Version-keyed cache invalidation

Analysis results are cached to disk keyed on two factors: the SHA-256 hash of the uploaded file and the analyzer version string (1.5.5). Upgrading to a new version of PyPRTG_CLA automatically invalidates all existing cached results and forces a fresh analysis — ensuring that improved parsing logic is always applied.

6. Comprehensive automated test suite

The application ships with 38 automated test functions covering:

  • Every data model contract and field type
  • Segment assignment and boundary detection
  • Error aggregation and pattern deduplication
  • Timeframe filtering across multiple segment configurations
  • All API endpoints (upload, progress, result, export)
  • Cache behavior and version-keyed invalidation
  • Status Data parser (HTML and JSON formats from support bundles)
  • Performance budgets: 5,000 errors processed in < 8 seconds; 200 restarts handled in < 5 seconds
  • Encoding handling (UTF-8, UTF-16, latin-1)
  • Score clamping (result always in 0–100)

These tests run against real parsing and rules logic using known input/output pairs with hardcoded expected values. If any test fails, the analyzer will not build.

7. Local-only processing

The application binds exclusively to 127.0.0.1. No uploaded file, no parsed result, and no exported report is ever transmitted to an external server. The machine running PyPRTG_CLA can be completely disconnected from the internet while still providing full functionality (charts load from a bundled vendor file in offline / export contexts).

Section 15

Troubleshooting

Browser cannot reach http://127.0.0.1:8077

Confirm the console window from PyPRTG_CLA.exe is still open and shows the Uvicorn running on http://127.0.0.1:8077 line. If it closed unexpectedly, relaunch the EXE. If it never appeared, check that the first-launch unpack completed (it can take up to 15 seconds).

Port 8077 is already in use

Another instance of PyPRTG_CLA (or another application) is already using port 8077. Close all PyPRTG_CLA.exe console windows. If the conflict persists, set the environment variable PRTG_ANALYZER_PORT to a free port number before launching, e.g. set PRTG_ANALYZER_PORT=9088 in a command prompt, then launch the EXE from that same prompt.

Slow first launch (5–15 seconds)

This is expected behavior. PyInstaller packs the entire Python runtime inside the EXE. On first run it extracts this runtime to a temporary folder. Subsequent launches from the same machine are significantly faster.

Windows Firewall prompt

You may safely allow local access when prompted. The app only binds to 127.0.0.1 and is never accessible from other machines on the network. Blocking the firewall prompt will prevent the browser from reaching the app.

Windows SmartScreen blocks the EXE

Click More info, then Run anyway. The binary is unsigned — code-signing certificates are expensive and this is an internal tool. The source code is included in the source/ folder if you wish to inspect or rebuild it. No data is sent to any external server.

Upload fails or analysis produces no results

  • Ensure the file is a valid PRTG Core.log (not Probe.log, WebServer.log, etc.).
  • Accepted extensions: .log, .gz, .txt.
  • Very short log files (e.g. newly started PRTG instances) may yield sparse results — this is normal.
  • Check the console window for any error messages from the server.
Section 16

Privacy & Security

  • Strictly local: The server binds to 127.0.0.1 only. No traffic from outside your machine can reach it. No data is transmitted externally.
  • No telemetry: PyPRTG_CLA makes no outbound network connections of any kind during normal operation.
  • License key masking: When you export an HTML report, the PRTG license key is automatically masked to show only the first 6 and last 4 characters (e.g. 001001…N47D). The raw key is never written to any exported file.
  • Cache location: Analysis results are cached in the OS temporary directory under prtg_analyzer_cache/. Files in this folder are keyed by the SHA-256 hash of the uploaded log and contain the parsed analysis data (not the raw log content). You can delete this folder at any time without affecting the application.
  • Memory resident for 1 hour: Active analysis jobs are kept in server memory for up to 1 hour after completion, then automatically evicted.
  • Source code included: The full Python source code, test suite, and build spec are included in the source/ folder. The application can be audited, rebuilt, or modified by anyone with a Python environment.
